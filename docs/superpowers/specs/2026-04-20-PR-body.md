# SessionRunner refactor: resilient session orchestration with BIDS layout, resume, and dry-run

**Spec:** `docs/superpowers/specs/2026-04-20-session-runner-design.md`
**Plan:** `~/.claude/plans/session-runner-implementation_eac93282.plan.md`

## Why

The old operator TUI was a forward-only wizard whose state lived in Python instance variables on three separate `Screen` subclasses. Five failure modes made it untrustworthy at the scanner:

1. **Brittle gating.** A completed step could not be redone; a RED step could not be advanced. Mid-session failures forced a full restart.
2. **No process resilience.** MURFI or PsychoPy dying mid-run orphaned the session.
3. **No resume.** `save_session_state` existed in `orchestration/subjects.py` but was never called.
4. **Missing protocols.** RT15 was not implemented; RT30's `NF_RUN_SEQUENCE` was missing its final `transferpost_run-02`.
5. **No meaningful dry-run.** `TestScreen` was a timer-based toy that could not rehearse RT15/RT30, crash recovery, or MURFI+PsychoPy coordination.

## What changed

A pure `SessionState` machine (frozen dataclasses, no I/O) + a `SessionRunner` imperative shell + a single `StepExecutor` Protocol that every step kind implements. Five concrete executors wrap the existing `orchestration/*` helpers without modifying them. The TUI becomes one `SessionScreen` parameterized by session type. Sessions are data (`mindfulness_nf/sessions.py`). BIDS-compliant filesystem layout; resume is automatic. Dry-run works out of the box — cacheless, with synthetic NIfTI/DICOM generation — and can optionally fetch real public BOLD data for realistic FSL rehearsal.

## Each failure mode, closed

| # | Old failure | New behavior |
|---|---|---|
| 1 | Forward-only, no retry | Navigation (`b`/`n`/`g`), restart (`r` — clear+retry), interrupt (`i`), resume from JSON. Cursor and execution decoupled. |
| 2 | Orphaned MURFI/PsychoPy | Per-executor subprocess ownership. `m` relaunches MURFI keeping progress and partial data via log byte-offset tracking; `p` relaunches PsychoPy. Status stays RUNNING while one component is down as long as others are healthy. |
| 3 | No resume | `SessionRunner.load_or_create` reads `session_state.json` (atomic, schema-versioned, self-contained config). Running-at-load-time steps coerce to `failed` with `error="interrupted by restart"`. |
| 4 | RT15 missing; RT30 truncated | `mindfulness_nf/sessions.py` defines all four session types: LOC3 (3 steps), RT15 (9), RT30 (15 — with the missing Transfer Post 2 restored), Process (7 FSL stages). |
| 5 | Toy dry-run | `--dry-run` launches real MURFI + real PsychoPy against a `SimulatedScannerSource` that synthesizes volumes at runtime (no cache needed) or uses a real public BOLD dataset via `scripts/fetch_dry_run_bold.py` for realistic FSL rehearsal. |

## Breaking changes

- **Old screens removed.** `LocalizerScreen`, `NeurofeedbackScreen`, `TestScreen` deleted. The unified `SessionScreen` replaces all three.
- **`SessionState` shape changed.** Old `steps: tuple[RunState, ...]` + `current_step: int` + `completed: bool` becomes `steps: tuple[StepState, ...]` + `cursor: int`; session completion is derived from `all(s.status == COMPLETED for s in steps)`. `RunState` is kept but repositioned as a view model for the existing widgets. Serialized sessions from the old model will not load (new schema_version=1 rejects missing frontmatter).
- **`NF_RUN_SEQUENCE` moved.** The module-level constant in `models.py` is gone; per-session config tuples live in `mindfulness_nf/sessions.py` (`LOC3` / `RT15` / `RT30` / `PROCESS`, keyword-arg constructors throughout).
- **Subject directory layout is BIDS.** New subjects land in `subjects/sub-NNN/ses-TYPE/{func,sourcedata/{murfi,psychopy},derivatives}/`. Legacy flat-layout helpers are still exported from `subjects.py` for back-compat, marked as deprecated.
- **New required runtime deps:** `hypothesis` (dev), `pytest-timeout` (dev), `nibabel` (runtime, used by synthetic_volumes), `nilearn` (optional via `[project.optional-dependencies].dry-run`).
